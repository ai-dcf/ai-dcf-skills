# 将想法脑暴成设计

## 概述

通过自然的协作式对话，把想法转化为完整成型的设计与规格说明。

先理解当前项目上下文，然后一次只问一个问题来逐步细化想法。当你已经理解要构建什么之后，把设计拆成多个小段（每段 200–300 字）呈现，并在每段后确认目前是否正确。

## 流程

**理解想法：**
- 先查看当前项目状态（文件、文档、最近提交）
- 一次只问一个问题来细化想法
- **反馈工具**：如果存在 `mcp-feedback-enhanced`（例如 `ask_followup_question`），用它来提问；如果没有，则用普通对话
- 能用选择题就优先用选择题，必要时也可以开放式提问
- 每条消息只问一个问题；如果某个话题需要深入，就拆成多个问题
- 聚焦理解：目的、约束、成功标准

**文档化与规格化：**
- **PRD（需求规格说明书）**：编写标准化的需求文档，包含背景、用户流程、功能详细说明、验收标准。
- **项目介绍（Project Introduction）**：编写面向利益相关者或用户的项目愿景、核心价值、Roadmap。
- **特性说明（Feature Docs）**：针对具体模块编写交互细节与逻辑说明。

**探索方案：**
- 提出 2–3 种不同方案并说明取舍
- 用对话式方式呈现选项，同时给出你的推荐与理由
- 先给出推荐选项，并解释原因

**呈现设计：**
- 当你确信自己理解要构建什么后，开始呈现设计
- 将内容拆成每段 200–300 字
- 每段结束后询问到目前为止是否正确
- 覆盖：架构、组件、数据流、错误处理、测试
- 如果有内容不清晰，随时回退并澄清

## 设计之后

**文档：**
- 将已验证的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果可用，使用 elements-of-style:writing-clearly-and-concisely 这个 skill
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问：“准备好进入实现阶段了吗？”
- 使用 superpowers:using-git-worktrees 创建隔离的工作区
- 使用 superpowers:writing-plans 生成详细的实现计划

## 关键原则

- **一次一个问题**：不要用多个问题让对方应接不暇
- **优先选择题**：在可能的情况下，比开放式问题更容易回答
- **彻底贯彻 YAGNI**：从所有设计中移除不必要的特性
- **探索替代方案**：在定案前总是提出 2–3 种方案
- **增量式验证**：分段呈现设计，并逐段确认
- **保持灵活**：当某些内容不清晰时，回退并澄清
